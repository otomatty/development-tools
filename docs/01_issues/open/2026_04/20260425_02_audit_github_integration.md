# GitHub 連携の現状調査

## 概要

UX 再設計の前提として、現在の GitHub 連携で「**何を・いつ・どの粒度で**」取得しているかを棚卸しする。これにより、活動可視化・ゲーミフィケーション・Issue 管理を強化するために不足しているデータと API 呼び出しを特定する。

## 背景・目的

- ホーム画面のゲーミフィケーション、Issue 管理、コントリビューショングラフはそれぞれ GitHub データを使っているが、データ取得経路と更新タイミングが分散している懸念がある
- UX 再設計（「今日のクエスト」「セッション計測」等）には、現在取得していないデータが必要になる可能性が高い
- 取得頻度・キャッシュ戦略・レート制限対応の現状把握なしに新機能を追加すると、API 制限到達や不整合の温床になる

## 調査対象

### 1. 認証

- [ ] OAuth 認証フロー (Device Flow + PKCE) の実装箇所
- [ ] トークン保存方法（暗号化方式、保存先）
- [ ] トークン有効期限・リフレッシュ戦略
- [ ] 必要なスコープ一覧（過不足の確認）

### 2. データ取得 API 呼び出し

各 Tauri コマンド・各バックエンド関数について以下を表にまとめる:

| コマンド/関数 | 呼び出す GitHub API | 取得するデータ | 呼び出しタイミング | キャッシュ有無 |
| ------------- | ------------------- | -------------- | ------------------ | -------------- |

調査対象（既知のものを最低限）:

- [ ] `get_github_stats` / `sync_github_stats`
- [ ] `get_contribution_graph`
- [ ] Issue 同期関連コマンド
- [ ] プロジェクト同期関連コマンド
- [ ] PR / レビュー関連の取得処理（XP 計算で使用しているはず）

### 3. データモデル・DB スキーマ

- [ ] `users`, `user_stats`, `xp_events`, `badges`, `streaks`, `challenges` 等のテーブル一覧
- [ ] それぞれの GitHub データとの対応関係
- [ ] 永続化されているデータと毎回 API を叩いているデータの区別

### 4. 同期戦略

- [ ] 自動同期のトリガー（起動時 / 定期 / 手動 / イベント駆動）
- [ ] 同期間隔の設定可能範囲（設定画面の「同期設定」と実装の整合性）
- [ ] バックグラウンド同期の有無と動作
- [ ] Stale-While-Revalidate 等のキャッシュ戦略の実装箇所
- [ ] レート制限到達時のフォールバック動作

### 5. XP / バッジ / ストリーク計算ロジック

- [ ] XP 計算ルール（コミット +10、PR +30、PR マージ +50、Issue 作成 +15、Issue 解決 +40、レビュー +25 が実装と一致しているか）
- [ ] バッジ判定ロジックの実装箇所と、判定タイミング（同期時 / リアルタイム）
- [ ] ストリーク計算の基準時刻（UTC / ローカル / GitHub 基準）
- [ ] 過去データの遡及計算が可能かどうか

### 6. エラーハンドリング・障害時の挙動

- [ ] 認証切れ時のユーザー体験
- [ ] ネットワーク断時の挙動
- [ ] レート制限到達時の表示・リトライ
- [ ] GitHub 側でリポジトリ削除等が発生した場合の整合性

## 成果物

`docs/02_research/2026_04/20260425_github_integration_audit.md` として以下を含むレポートを作成する:

1. **現状サマリ** — どこまで実装されているか一覧
2. **API 呼び出しマップ** — 上記の表
3. **データフロー図** — GitHub → Tauri command → DB → UI の経路
4. **同期戦略の現状** — トリガー・頻度・キャッシュ
5. **ギャップリスト** — UX 再設計で必要になるが未取得 / 未実装のもの
   - 例: 「今日のコミット数のリアルタイム取得」「未着手 Issue 一覧の優先度付き取得」
   - 各項目について新規 API 呼び出しの要否、レート制限影響度、実装難易度を併記
6. **リスク・課題** — レート制限、整合性、トークン管理等の懸念点

## 受け入れ条件

- [ ] 上記成果物ドキュメントが作成されている
